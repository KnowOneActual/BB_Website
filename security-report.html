Website Technology & Security Enhancements

This page provides an overview of the technologies and practices that power my personal website, focusing on how I've enhanced its security, performance, and overall user experience.

The Technology Stack

My website is built with a modern and robust technology stack designed for performance, security, and scalability.

  • Frontend: HTML5, CSS3, JavaScript (Vanilla JS, Chart.js for data visualization, Three.js for interactive hero background).
  • Styling: Tailwind CSS (utility-first CSS framework for rapid and responsive design).
  • Hosting & Serverless: Netlify (for seamless deployment, continuous integration, and serverless functions).
  • Serverless Functions (Node.js):
    • Weather Bot: Integrates with Google Gemini API (for AI/NLP) and OpenWeatherMap API (for weather data).
    • Contact Form: Utilizes Resend for secure email delivery and Cloudflare Turnstile for CAPTCHA verification.
  • Database: Firebase Firestore (for real-time chat history in the Weather Bot).

Key Improvements & Actions Taken:

1. Robust HTTP Security Headers (_headers Configuration) & Security Policy:

Action: I implemented a comprehensive `_headers` file for Netlify, meticulously defining crucial HTTP security headers. Additionally, I established a `security.txt` file at `/.well-known/security.txt` to provide a clear vulnerability reporting channel.

Why it matters: This setup provides a robust defense by:

  • Content-Security-Policy (CSP): Establishing strict rules to prevent Cross-Site Scripting (XSS) and data injection attacks. By whitelisting all allowed sources for scripts, styles, images, and other content, I significantly reduced the risk of malicious code execution.
  • X-Frame-Options (DENY): Preventing clickjacking by disallowing my site from being embedded in iframes on other domains.
  • X-Content-Type-Options (nosniff): Mitigating MIME-sniffing attacks, which helps prevent browsers from misinterpreting file types.
  • Referrer-Policy (no-referrer-when-downgrade): Enhancing user privacy by controlling how much referrer information is sent with requests.
  • Permissions-Policy: Explicitly disabling unwanted browser features (like geolocation, microphone, camera) to enhance user privacy and security.
  • security.txt: This standardized file demonstrates a proactive stance on security, offering a designated and easily discoverable channel for security researchers to report vulnerabilities responsibly. It highlights a commitment to security best practices and responsible disclosure.

2. Code Refactoring for Enhanced Security & Maintainability:

Action: I moved all inline JavaScript and CSS from the `index.html` file into dedicated external files (`script.js` and `style.css`). Additionally, JavaScript event handlers (e.g., `onsubmit`, `onclick`) were refactored to be attached programmatically in `script.js`.

Why it matters: This change was crucial. It allowed me to remove the potentially dangerous `'unsafe-inline'` directive from the CSP for my custom code, significantly reducing the XSS attack surface. It also greatly improved code organization, readability, and overall maintainability.

3. Major Performance Optimization (Removing Unnecessary Payload):

Action: A critical performance issue was identified: an unintended 4.1 MB text file (`block.txt`) was being loaded from GitHub on every page visit. I successfully eliminated this reference.

Why it matters: This removal dramatically reduced the total page size from nearly 5 MB to less than 1 MB. This directly translated to much faster loading times and a significantly better user experience for visitors, especially on mobile devices.

4. Accessibility Enhancements:

Action: I addressed color contrast issues in the footer text by adjusting its color for better readability on a dark background. I also implemented the <main> HTML5 semantic tag to properly structure the main content of the page.

Why it matters: These improvements ensure my site is more accessible to users with visual impairments and assistive technologies, providing a better experience for all visitors.

5. Favicon Reliability:

Action: I updated the favicon link to point to a standard `favicon.ico` file.

Why it matters: This resolved an `ERR_NAME_NOT_RESOLVED` console error previously seen in Lighthouse, ensuring the favicon loads reliably.

Why Some Items Were Not Fully Addressed (Practical Considerations):

  • 'unsafe-inline' in CSP (Persistent Warning): While I eliminated `'unsafe-inline'` for my custom code, it remains in the CSP for specific third-party integrations like Google reCAPTCHA and Tailwind CSS (from their CDN). These services dynamically inject small amounts of inline script and style. Implementing CSP nonces or hashes for such dynamically generated third-party resources is technically complex for a static site hosted on Netlify, often requiring server-side logic. For a personal portfolio, the robust domain whitelisting in my CSP provides strong protection, making this a pragmatic and acceptable compromise for functionality.
  • Subresource Integrity (SRI) for Tailwind CSS: My attempts to add SRI to the Tailwind CSS CDN link were unsuccessful because the CDN's Cross-Origin Resource Sharing (CORS) headers are not configured to support SRI requests. As this is a server-side limitation of the CDN, it cannot be fixed from my end. However, the explicit whitelisting of `cdn.tailwindcss.com` in my CSP still ensures that only the legitimate Tailwind CSS from that trusted domain can be loaded.
  • Absence of Anti-CSRF Tokens: My contact form utilizes Formspree, a third-party service that typically handles CSRF protection on their backend. For a simple contact form that doesn't process sensitive data, this is an acceptable delegation of responsibility to a trusted service.
